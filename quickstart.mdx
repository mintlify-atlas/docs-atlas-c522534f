---
title: "Quickstart"
description: "Deploy a PostgreSQL cluster on a local Kubernetes cluster in minutes using Kind or Minikube."
---

This guide walks you through testing a PostgreSQL cluster on your local machine by deploying CloudNativePG on a local Kubernetes cluster using either [Kind](https://kind.sigs.k8s.io/) or [Minikube](https://kubernetes.io/docs/setup/learning-environment/minikube/). By the end, you will have a running 3-node PostgreSQL cluster you can experiment with.

<Warning>
  These instructions are for demonstration, testing, and practice purposes only. Do not use in production.
</Warning>

<Info>
  Make sure you have `kubectl` installed on your machine before proceeding. See the Kubernetes documentation on [how to install kubectl](https://kubernetes.io/docs/tasks/tools/install-kubectl/).
</Info>

<Steps>
  <Step title="Set up a local Kubernetes cluster">
    Choose either Kind or Minikube to create a local Kubernetes cluster.

    <Tabs>
      <Tab title="Kind">
        Kind (Kubernetes IN Docker) runs local Kubernetes clusters using Docker container nodes. Install `kind` by following the [Kind quickstart](https://kind.sigs.k8s.io/docs/user/quick-start), then create a cluster:

        ```bash
        kind create cluster --name pg
        ```
      </Tab>
      <Tab title="Minikube">
        Minikube runs a single-node Kubernetes cluster inside a VM on your laptop. Install Minikube by following the [Minikube installation guide](https://kubernetes.io/docs/tasks/tools/install-minikube), then start a cluster:

        ```bash
        minikube start
        ```

        Verify the cluster is running:

        ```bash
        kubectl get nodes
        ```

        You will see one node called `minikube`.
      </Tab>
    </Tabs>
  </Step>

  <Step title="Install CloudNativePG">
    Apply the CloudNativePG operator manifest to your cluster:

    ```bash
    kubectl apply --server-side -f \
      https://raw.githubusercontent.com/cloudnative-pg/cloudnative-pg/release-1.29/releases/cnpg-1.29.0.yaml
    ```

    Verify the operator deployment is ready:

    ```bash
    kubectl rollout status deployment \
      -n cnpg-system cnpg-controller-manager
    ```

    The operator is deployed in the `cnpg-system` namespace as a Deployment named `cnpg-controller-manager`.
  </Step>

  <Step title="Deploy a PostgreSQL cluster">
    To deploy a PostgreSQL cluster, apply a manifest that defines your desired `Cluster` resource. Create a file called `cluster-example.yaml` with the following content:

    ```yaml cluster-example.yaml
    apiVersion: postgresql.cnpg.io/v1
    kind: Cluster
    metadata:
      name: cluster-example
    spec:
      instances: 3
      storage:
        size: 1Gi
    ```

    This defines a 3-instance cluster using the default storage class with 1Gi of storage per instance.

    Apply the manifest to create the cluster:

    ```bash
    kubectl apply -f cluster-example.yaml
    ```

    Check that the pods are being created:

    ```bash
    kubectl get pods -l cnpg.io/cluster=cluster-example
    ```

    <Note>
      The operator labels all objects belonging to a cluster with `cnpg.io/cluster`. Use this label to filter resources for a specific cluster.
    </Note>

    By default, the operator installs the latest available minor version of the latest major version of PostgreSQL at the time of the operator release. To use a specific PostgreSQL version, set the `imageName` field in the `spec` section:

    ```yaml
    apiVersion: postgresql.cnpg.io/v1
    kind: Cluster
    metadata:
      name: cluster-example
    spec:
      instances: 3
      imageName: ghcr.io/cloudnative-pg/postgresql:13.6
      storage:
        size: 1Gi
    ```

    <Warning>
      Always point to a specific image version. Never use tags like `latest` or `13` in a production environment, as they can lead to unpredictable update behavior and version inconsistency across the cluster. For fully deterministic deployments, use the `<image>:<tag>@sha256:<digestValue>` format.
    </Warning>
  </Step>

  <Step title="Set up monitoring (optional)">
    <Info>
      Installing Prometheus and Grafana is beyond the scope of this project. The instructions in this step are provided for experimentation and illustration only.
    </Info>

    CloudNativePG exposes metrics that can be scraped by Prometheus. This step uses the [Kube-Prometheus stack](https://github.com/prometheus-community/helm-charts/tree/main/charts/kube-prometheus-stack) Helm chart, which includes the Prometheus Operator, Alert Manager, and Grafana.

    If you do not have [Helm](https://helm.sh) installed, follow the [Helm quickstart](https://helm.sh/docs/intro/quickstart/) to install it first.

    Add the `prometheus-community` Helm repository and install the Kube Prometheus stack:

    ```bash
    helm repo add prometheus-community \
      https://prometheus-community.github.io/helm-charts

    helm upgrade --install \
      -f https://raw.githubusercontent.com/cloudnative-pg/cloudnative-pg/main/docs/src/samples/monitoring/kube-stack-config.yaml \
      prometheus-community \
      prometheus-community/kube-prometheus-stack
    ```

    After installation, deploy a cluster with monitoring enabled:

    ```bash
    kubectl apply -f - <<EOF
    ---
    apiVersion: postgresql.cnpg.io/v1
    kind: Cluster
    metadata:
      name: cluster-with-metrics
    spec:
      instances: 3
      storage:
        size: 1Gi
    ---
    apiVersion: monitoring.coreos.com/v1
    kind: PodMonitor
    metadata:
      name: cluster-with-metrics
    spec:
      selector:
        matchLabels:
          cnpg.io/cluster: cluster-with-metrics
      podMetricsEndpoints:
      - port: metrics
    EOF
    ```

    **Access Prometheus**

    Port-forward the Prometheus service and open [http://localhost:9090](http://localhost:9090) in your browser:

    ```bash
    kubectl port-forward svc/prometheus-community-kube-prometheus 9090
    ```

    You can also apply the default CloudNativePG alert rules:

    ```bash
    kubectl apply -f \
      https://raw.githubusercontent.com/cloudnative-pg/cloudnative-pg/main/docs/src/samples/monitoring/prometheusrule.yaml
    ```

    **Access Grafana**

    Port-forward the Grafana service and open [http://localhost:3000](http://localhost:3000) in your browser:

    ```bash
    kubectl port-forward svc/prometheus-community-grafana 3000:80
    ```

    Log in with username `admin` and password `prom-operator`. To add a CloudNativePG dashboard, download [grafana-dashboard.json](https://github.com/cloudnative-pg/grafana-dashboards/blob/main/charts/cluster/grafana-dashboard.json) from the [`grafana-dashboards` repository](https://github.com/cloudnative-pg/grafana-dashboards) and import it via **Dashboards > New > Import**.

    <Warning>
      Some graphs in the CloudNativePG Grafana dashboard use metrics that may be in alpha stage, such as `kubelet_volume_stats_available_bytes` and `kubelet_volume_stats_capacity_bytes`, which can result in some panels showing "No data".
    </Warning>
  </Step>
</Steps>
